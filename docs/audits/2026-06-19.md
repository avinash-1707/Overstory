# Data-flow audit — 2026-06-19

Whole-app audit of Overstory (no target → exhaustive across every flow). Method: 5 discovery
scouts (one per area) → 6 clustered deep audits (`sage`, end-to-end through validation, error
handling, edge conditions, data integrity, security, policy↔enforcement, concurrency, data
lifecycle, trust boundaries) → adversarial re-verification of every Critical/High by direct
re-read of the cited code.

**Scope note:** coverage is complete — all ~24 in-scope flows across `apps/{api,web}` and
`packages/{core,db,cli,mcp,config}` were audited. One discovery scout (apps/web) returned a
stale "incomplete build" result; its slice was re-mapped and audited directly, so no flow was
skipped. Not exercised at runtime (static audit only): the `hashtext` advisory-lock collision
rate and Postgres `'Infinity'::float` sort behavior — both flagged for a unit test once the
ranking clamp lands.

## Findings

### CRITICAL

**C1 — Cross-tenant data exposure via session-blind dashboard + open sign-up.**
`apps/web/src/server/dashboard.ts:13-24` (`resolveDashCtx` → `organization LIMIT 1`) +
`packages/core/src/auth/index.ts:29` (`emailAndPassword: { enabled: true }`, no
`disableSignUp`) + `apps/web/src/server/auth.functions.ts:15-19` (`requireSession` accepts
*any* valid session). These three compose: anyone who can reach the web origin self-registers
at `/sign-up`, passes the gate, and `resolveDashCtx` hands them the single seeded (dogfood)
workspace regardless of who they are. A self-registered stranger reads the full sessions list,
every session timeline (decision title/statement/rationale `why`/file paths via
`loadDecisionViews`), and activity metrics including `guardByPath` (which leaks the repo's file
layout). For a tool whose product *is* the private "why" behind a codebase, that is the worst
payload to leak.
- **Real now, gated by deployment.** It is a live breach the moment `:3000` is reachable by
  anyone but the operator. If the web origin is bound to localhost during the dogfood window,
  practical blast radius is small — but the code ships the hole.
- **Fix (Quick, do before any non-local exposure):** set `emailAndPassword.disableSignUp: true`
  and seed the operator account out-of-band. That converts C1 from live exposure to latent.
  Then land the real fix with D36: derive workspace from `session.activeOrganizationId` (member
  fallback), or gate `requireSession` on membership of the seeded org and 403 otherwise. The
  `D34 analog` comment at `dashboard.ts:8-12` already anticipates this; the code hasn't shipped it.

### HIGH

**H1 — Auth/DB path can hang on an unreachable database (availability, NOT an auth bypass).**
`packages/db/src/client.ts:17` creates `postgres(connectionString)` with no `connect_timeout`
or statement timeout; `apps/api/src/middleware/auth.ts:24-33` has no try/catch; `apps/api/src/index.ts`
has no `app.onError`. On a thrown DB error the request 500s (fail-closed — correct). On a
network black-hole the lookup can hang for the OS TCP timeout (many seconds to minutes) before
rejecting. Under load the event loop fills with pending auth lookups. **Important calibration:**
the gate never fails *open* — `c.set('auth')`/`next()` are unreachable on throw or hang, so no
request proceeds unauthenticated. This is resilience/availability, not a security bypass.
- **Fix (Quick):** `postgres(url, { connect_timeout: 5 })` plus a statement timeout; add an
  explicit `app.onError` returning a bodyless 503. Makes the gate fail *fast*-closed, never hang.

**H2 — No application-level rate limiting or per-key quota on the machine API; the metered
`/v1/mcp/check` path is an unbounded cost sink.** `apps/api/src/index.ts:23` mounts only
`apiKeyAuth` on `/v1/*` — no throttle. Two surfaces:
(a) every invalid bearer triggers a full indexed DB lookup (cheap brute/DoS amplifier);
(b) `POST /v1/mcp/check` (`apps/api/src/routes/mcp.ts:53-67`) is the **only path to real dollars**
— each call is individually bounded (`maxTokens 2000`, 50s, `FILE_MATCH_LIMIT 50`) but the call
*count* is not. A valid-but-runaway/compromised key spends OpenRouter credits until the account
drains (then degrades open). The credit ledger is explicitly deferred (D32), so today the only
backstop is the 402-fail-fast. `/api/auth/*` brute-force relies on Better Auth's defaults, which
are off in dev and in-memory-only in prod.
- **Fix (Short):** coarse per-`apiKeyId` token bucket on `/v1/mcp/check` (in-memory is enough
  for dogfood — `serve_events` already logs every call to derive a count); IP rate-limit `/v1/*`
  *before* the DB lookup; enable Better Auth's `rateLimit` and decide whether `/api/auth/*`
  should exist on the api origin at all (D29 says api is machine-only → likely remove that mount,
  which also shrinks the shared-secret blast radius).

**H3 — Capture ranking integrity: `composite` is client-supplied and unclamped, yet drives the
always-on tier and the serving sort.** `apps/api/src/routes/capture.ts:17-22` (`rankingSchema`,
bare `z.number()`), `:111` (`alwaysOn = d.ranking.composite >= 0.66`), consumed by
`getAlwaysOn`/`searchByQuery` `ORDER BY (ranking->>'composite')::float desc`. The `:14` comment
claims `alwaysOn` is "derived server-side" — misleading: only the *threshold* is applied
server-side; the *value* is trusted verbatim. A buggy/hostile CLI setting `composite: 1.0` puts
every decision into the always-on context injected into every agent session. Unclamped also
accepts `NaN`/`Infinity` — Postgres parses `'Infinity'::float`, corrupting sort order;
`NaN >= 0.66` is silently false. "Client" here is the owner's own ApiKey-gated CLI, so this is
an integrity/robustness finding, not external attack — but the always-on tier is the highest-
blast-radius surface in the product.
- **Fix (Short):** recompute `composite` server-side via `@overstory/core/capture/ranking`'s
  `composite()` from the three sub-signals; clamp each sub-signal to `[0,1].finite()`; drop the
  client `composite` from the schema. Makes the comment true.

**H4 — No size/count bounds on the capture payload (unbounded-write DoS).**
`apps/api/src/routes/capture.ts:24-41`: `decisions` has `.min(1)` but no `.max()`; `title`/
`statement`/`rationale.why` have no length cap; `pointers`/`topics`/`alternativesConsidered`
and their items are uncapped; `flow.name`/`description`/`resolvedSha` uncapped. One authed POST
with a huge array or a multi-MB `statement` → unbounded DB growth, memory blowup building
`pointerRows`, oversized rows, and a giant always-on payload later shipped to every agent.
ApiKey-gating bounds *who*, not *how much* — and the content originates from LLM output
(untrusted-content path).
- **Fix (Quick):** add `.max()` to every string/array in `bodySchema`/`decisionSchema`
  (e.g. `decisions.max(200)`, `statement.max(4000)`, `pointers.max(100)` + item `.max(1024)`, …).

**H5 — No explicit `trustedOrigins` on the shared Better Auth config (CSRF backstop).**
`packages/core/src/auth/index.ts:24-34` sets `baseURL` but no `trustedOrigins`, while D37 runs
two origins (web :3000, api :3001) on one secret + DB. Better Auth's default origin check uses
`baseURL` only, so you rely on the implicit default + SameSite cookie defaults rather than an
explicit allowlist; cross-origin session flows between the two origins are also unspecified.
- **Fix (Short):** `trustedOrigins: [WEB_BASE_URL, BETTER_AUTH_URL]` read from `@overstory/config`.
  Explicit is correct precisely because two origins share one secret.

**H6 — `overstory_check`'s 60s timeout is uncoordinated with the MCP client's own request
timeout.** `packages/mcp/src/tools/index.ts:78` passes `timeoutMs = 60_000` for `check`. That is
the *server-proxy* abort ceiling; the coding-agent's MCP client has its own request timeout
(spec default ~60s, often lower / host-set). If the backend answers at 45-58s, the client may
cancel first and surface a timeout error for `overstory_check` — exactly the "Overstory
blocked/annoyed me" outcome the degrade-open invariant forbids, on the one tool most likely to
be slow. The backend never degrades open on its own deadline. (Same root cause as the LLM
wall-clock finding M3 below — fix them together.)
- **Fix (Short):** cap the server-side LLM judgment well under the client floor (treat ~25-30s
  as hard ceiling); have the *backend* return an empty/annotated `check` result if its LLM call
  exceeds ~25s; set the MCP `call()` timeout for `check` a hair above that (~30s), not 60s.

### MEDIUM

- **M1 — CLI surfaces raw backend/LLM response bodies to the user; reflected body could echo the
  Bearer key.** `packages/cli/src/index.ts:69` (`persist()` throws `…: ${await res.text()}`) and
  `packages/core/src/llm/index.ts:172,177` (`LlmError` embeds `res.text()`), both printed via the
  top-level `.catch` at `index.ts:280`. The key value is never *directly* printed, but if any
  backend/proxy reflects the request (debug/500 traces) the `OVERSTORY_API_KEY`/`OPENROUTER_API_KEY`
  lands in the user's terminal + scrollback. **Fix:** in the top-level catch, redact known secrets
  (`msg.replaceAll(env.OVERSTORY_API_KEY ?? '\0','[redacted]')…`) and keep raw bodies behind a debug flag.
- **M2 — MCP server logs the raw error object to stderr** (`packages/mcp/src/index.ts:33`,
  `tools/index.ts:124,129`). Not channel-corrupting (stderr ≠ JSON-RPC) and undici doesn't echo
  request headers, but `err` is logged whole and shapes are version-dependent. **Fix:** log
  `err instanceof Error ? err.message : String(err)`, not the object.
- **M3 — Total LLM wall-clock can exceed the 60s MCP patience.** `packages/core/src/llm/index.ts:158-203`
  retries up to 4× with a *fresh* 50s timer each (lib/llm.ts:30) + the reasoning-fallback adds
  another full `send()`. The agent is still unblocked (its `call()` aborts at 60s → degrade-open),
  but the backend keeps spending on a discarded result and logs misleading latency. **Fix:** bound
  *total* time — for the check path set `maxRetries: 1` and `timeoutMs ≈ 25s`, and don't retry
  `AbortError` (`index.ts:194-195` treats a timeout as retryable).
- **M4 — Check prompt input is uncapped.** `packages/core/src/contradiction/index.ts:49-52`
  interpolates each candidate's full `statement`+`why`; `FILE_MATCH_LIMIT` caps candidate *count*
  (50) but not content length → large input-token cost on the metered path. **Fix:** truncate each
  field (`.slice(0,1000)`) in `buildPrompt`.
- **M5 — A decision with zero servable pointers persists silently.** `capture.ts:121-130`: if all
  paths normalize to empty, `pointerRows` is empty but `persisted++` still runs → a decision
  `guard` can never surface (servable only via context/search). **Fix:** return a `pointerless`
  count so the CLI can warn, or reject malformed (non-empty in, all-empty out) pointer lists.
- **M6 — CLI loses the human's interactive answers on persist failure.** `index.ts:266-271`: a
  transient 500 from `/v1/capture` throws away N answered provocations (the JSON-dump branch only
  fires when env is *unset*, not on failure). The server txn correctly rolls back atomically — this
  is pure UX. **Fix:** on persist failure, dump `captured` to a file/stdout.
- **M7 — Web `createAuth` omits the `tanstackStartCookies` plugin** that `core/auth/index.ts:30-33`
  says the web "adds … which must come LAST." `apps/web/src/server/auth.ts:9` passes no `plugins`.
  Sign-in likely works because `/api/auth/$` forwards the raw `Set-Cookie` Response, but the
  documented invariant is violated and future cookie-setting server fns will break. **Fix:** pass
  `{ plugins: [tanstackStartCookies()] }` or delete the misleading comment after verifying end-to-end.
- **M8 — Unbounded `getSessionTimeline` result.** `packages/core/src/dashboard/index.ts:125-154`
  has no `limit` (unlike `listSessions`); a long-lived session's full event set + per-id fan-out is
  materialized into React. Latency/memory cliff under real traffic. **Fix:** add a `limit` (~500,
  paginated).
- **M9 — No revocation path for an ApiKey beyond expiry or row-delete.** `packages/db/src/schema/api-key.ts`
  has `expiresAt` but no `revokedAt`/`active`. Killing a leaked key *now* means DELETE (loses audit
  trail). **Fix:** add nullable `revokedAt`; check it alongside the expiry test in `auth.ts:36`.
- **M10 — `last_used_at` write on every authed request.** `apps/api/src/middleware/auth.ts:43-49`
  issues a row UPDATE per request (correctly detached + error-swallowed). A chatty MCP session
  produces a write storm on one hot row + MVCC bloat on Neon. **Fix:** only update when
  `last_used_at` is older than ~60s.

### LOW

- **L1 — Non-deterministic tie-break at the serving limits.** `serving/index.ts:117,167,220` order
  by composite with no secondary key → two equally-ranked decisions can flap in/out of the agent's
  context across calls at the 50/50/10 boundary. **Fix:** add `decisions.id` as a final `orderBy`.
- **L2 — `normalizePath` doesn't strip `..` or normalize case.** `packages/core/src/paths.ts:4-10`.
  Not a tenant leak (matching is repo-scoped); worst case a silent non-match on case-variant or
  traversal-shaped input. Capture-write and serving-read use the *same* function (agreement
  **confirmed**), so any fix applies to both sides automatically.
- **L3 — `readFlowFiles` has no path containment / follows symlinked files.** `packages/cli/src/files.ts`.
  Symlinked *dirs* aren't recursed (accidental containment); a symlinked *file* with an allowed
  extension is read. Acceptable under the dogfood trust model (CLI runs on owner's machine); harden
  with a `resolve().startsWith(repoRoot)` check + `lstat` if ever run against untrusted input. Also:
  the 24KB cap slices by UTF-16 length, not bytes (cosmetic).
- **L4 — Oversized Authorization header is hashed/looked-up before rejection.** `lib/api-key.ts:33`.
  Node's ~16KB header limit mostly caps this. **Fix (optional):** reject `token.length > 200` pre-hash.
- **L5 — `--budget` has no upper bound** (`cli/src/index.ts:187-191`); effective cap is candidate
  count. **L6 — `requireSession` throws bare `Error('Unauthorized')`** (`auth.functions.ts:17`) →
  opaque server-fn error instead of a 401/redirect on client-side revalidation after expiry.

## Verified sound (no change needed)

The core invariants hold and are adversarially robust:

- **Tenant boundary (D34).** Every serving query scopes `decisions.repoId = ctx.repoId`;
  `pointers` (no `repo_id`) is reached only via repo-scoped `decision_id`. `getDecision` ANDs
  `repoId`, so a foreign decision id returns null/404 — confirmed not a cross-tenant read. `ctx`
  derives from the ApiKey row, never client input.
- **The raw-SQL / Drizzle bare-column gotcha is correctly dodged.** Traced byte-by-byte:
  `POINTERS_AGG` qualifies its columns (both tables in the top-level join); the `exists()` in
  `findDecisionsByFiles` uses `alias()` + `eq()` correlation so `${decisions.id}` emits the
  qualified outer reference, not the bare `"id"` that would silently match nothing.
- **The empty-files invariant is load-bearing and unbypassable.** `findDecisionsByFiles:139`
  early-returns `[]` when normalized is empty — without it `or(...[])` collapses the `exists()`
  WHERE to match every decision in the repo. Both callers (guard, check) route through it.
- **`starts_with` directory match is correct** — `starts_with(file, ptr||'/')` matches
  `src/auth/login.ts` under pointer `src/auth` but not `src/authentication.ts` (the `/` boundary).
  (A reviewer flagged then retracted this.)
- **Degrade-open is defense-in-depth and complete.** `check` has three catch layers (no-key,
  outer route, internal); `findDecisionsByFiles` throwing *before* the internal try is still
  caught by the route. The MCP `call()` never throws on any path (timeout, non-ok, 404, network,
  JSON-parse), and the MCP SDK itself converts any handler throw into an `isError` tool result —
  the JSON-RPC channel cannot be corrupted.
- **Hallucination guard is airtight.** `byId.get(v.decisionId)` value-compares against the
  repo-scoped candidate ids; the `as DecisionId` cast is compile-time only. The model literally
  cannot surface a decision outside the prompt's candidate set. Prompt-injection via `summary`
  is bounded to self-inflicted (same agent writes and reads), and cross-tenant exfiltration is
  closed.
- **Warn-only holds.** `check`'s only write is the `serve_events` row; it never flips
  `decisions.status` → no false freshness (the flip belongs to D27's PR-time backstop).
- **402 fails fast and degrades open** (not in `RETRYABLE`); secret hygiene in the LLM client is
  clean (key only in the Authorization header, never in logs/errors/`onUsage`).
- **Append-only + idempotency (D11).** Advisory lock serializes same-flow captures; dedup on
  `(flowId, statement, capturedFrom)` (intra-batch dups caught via in-txn read-your-writes);
  `onConflictDoNothing` on flows/pointers. Atomic rollback on mid-loop failure confirmed.
- **stdout hygiene, session-id non-spoofability, decision-id `encodeURIComponent`, FTS bound-param
  injection-safety** — all confirmed.

## Verdict

The product's hard invariants — the multi-tenant boundary, append-only capture, degrade-open
serving, the LLM hallucination guard, warn-only check — are **correctly and robustly
implemented**, and the most error-prone spot (the raw-SQL pointer hydration) is handled and
accurately commented. The gaps are at the operational edges, and they cluster into three themes:

1. **The web dashboard is not yet tenant-safe (C1).** Session-blind `resolveDashCtx` + open
   sign-up is a real cross-tenant leak the moment the web origin is exposed. Close it now with
   `disableSignUp: true`; fix properly with D36.
2. **No throttling anywhere (H2, partly H1).** No rate limiting, no per-key quota, no spend cap on
   the metered LLM path, and a DB-hang risk on the hot auth path. All independently surfaced by
   two clusters.
3. **Capture trusts its client too much (H3, H4).** Client-supplied ranking drives the always-on
   tier; no input bounds. Both are Quick schema-level fixes.

Recommended order: **C1 (`disableSignUp`)** → **H1 (DB timeouts + `onError`)** → **H4 (input
bounds)** + **H3 (recompute/clamp ranking)** → **H6/M3 (coordinate check timeouts)** → **H2
(rate limit the check path)** → **H5 (`trustedOrigins`)** → Mediums. Most are Quick/Short; none
require architectural change.

## Resolution (2026-06-20)

All findings closed. **C1 + H1–H6** in `28b53a2`, **M1–M10** in `283b7f6`, **L1–L6** in this
batch (serving tiebreak, `normalizePath` traversal-collapse, CLI symlink/containment + byte-clip,
bearer length cap, `--budget` upper bound, `requireSession` → router redirect). The deep fixes
(tenant safety, metered spend) remain tracked as D36 (multi-tenant) and D32 (credit ledger).
