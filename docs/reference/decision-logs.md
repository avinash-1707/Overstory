# Overstory — Decisions Log

> **Status:** Living document, append-only. Every meaningful project decision lands here with its reasoning, so future-you (or a collaborator) never has to reconstruct _why_ a cut was made. Do not delete entries — supersede them.

---

## How to use this log

- **Append, don't edit.** When a decision changes, add a new entry that references and supersedes the old one. The history is the point.
- Each entry: what was decided, why, and (where relevant) what it deliberately rules out.
- This is Overstory's own "decision memory" — we eat our own dog food.

## Foundational decisions (D1–D9)

These came out of the idea-shaping phase and define the product's shape.

| #      | Decision                                                            | Why                                                                                                                                                              | Rules out                                                   |
| ------ | ------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------- |
| **D1** | Don't store code; store rationale                                   | Agents fetch live code well; caching it is stale + pointless. The unowned value is the "why" not in syntax.                                                      | Code cache, syntax storage                                  |
| **D2** | Don't compete on analysis accuracy                                  | Greptile-grade analysis is unwinnable cold against a funded incumbent.                                                                                           | Deep static analysis, race-condition detection as a feature |
| **D3** | Primary value = preserving intentional design, not raw token-saving | The real, defensible pain is agents re-making banned decisions. This also dodges the trust-on-read problem — we claim _memory of decisions_, not _intelligence_. | "Token-saving" as the headline pitch                        |
| **D4** | Capture via provocation-and-defense, not documentation              | Devs skip documentation chores. A review where rejection = confirmation turns capture into engagement; rationale arrives as a byproduct of defending a choice.   | "Please document your architecture" flows                   |
| **D5** | Suggestions aim to provoke, not to be correct                       | Keeps the quality bar winnable and keeps us off the analysis arms race. Frame as options/tradeoffs, never "the right answer."                                    | Suggestion engine that must be accurate                     |
| **D6** | Staleness is largely dodged because rationale is stable             | Decisions change on big events, not every PR. We persist exactly the knowledge that doesn't churn. (Caveat: silent erosion — see D10.)                           | Treating decisions as volatile as code                      |
| **D7** | Invalidate only on decision change, not every diff                  | Because rationale is stable, expensive re-derivation is rare. This is what makes the economics work.                                                             | Re-analyzing on every commit                                |
| **D8** | Onboarding pass + incremental passes                                | Heavy architecture review is occasional. Build foundational memory once; grow it per new flow/PR.                                                                | Continuous full-repo re-review                              |
| **D9** | One language/ecosystem first                                        | Depth over breadth. Be genuinely good for one stack before generalizing.                                                                                         | Multi-language MVP                                          |

## Decisions added during stress-testing

| #       | Decision                                                                                                                      | Why                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| ------- | ----------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **D10** | Design choices are captured at **first ingestion**; suggestions to change only appear if the user _wants_ to change something | If a user keeps an unconventional choice, that _is_ a recorded design decision — equally valid. We record decided/not-decided, not good/bad. This resolves the "suggestion must be correct" worry.                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| **D11** | Freshness signal = **contradiction events**, not erosion detection                                                            | Silent decision-erosion has no diff to detect. Instead of trying to detect erosion directly, trigger re-confirmation when an agent or PR _contradicts_ a stored decision ("this change goes against a recorded decision — still true?"). Tractable; turns the unsolvable into an event-driven check.                                                                                                                                                                                                                                                                                                                                          |
| **D12** | Subagents scoped per-flow, not whole-software                                                                                 | Avoids over-generalizing the architecture. Each subagent reasons about one flow/feature, making inferences accurate at capture time.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| **D13** | Dogfood first on Kakiyo; bar = "do I keep using it"                                                                           | Self-use parks the unproven buyer question and gives the most honest engagement signal (no politeness bias). If the most motivated user won't feed the loop, no buyer would.                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| **D14** | Product name: **Overstory**                                                                                                   | In forestry, the overstory is the topmost forest layer that governs everything below — sits above the code, preserves intentional design, agent works underneath. Sharper than "Canopy" (the precise term for the canopy layer) and more ownable.                                                                                                                                                                                                                                                                                                                                                                                             |
| **D15** | Store **pointers to code**, not code itself                                                                                   | A decision carries a reference (file paths / symbols / anchors) to where it applies, so the agent jumps straight there instead of burning tokens _searching_ the repo to locate it. This is the token-saving lever — eliminating the search, not caching the bytes. Crucially this does NOT reverse D1: we store the _address_, not the _contents_. The agent reads live code at the pointer (never stale). Preserves D2/D6/D7 because addresses are stable and file-moves are rare + diff-detectable, unlike code contents. Cost: light pointer-drift maintenance (re-resolve on file rename/move — a cheap, detectable invalidation event). |
| **D16** | **Embrace deep analysis as the internal capture engine** (refines D2, does not supersede it)                                  | Credible provocation (D5) requires genuinely understanding a flow — a naive suggestion poisons trust (provocation-quality watch-item). The bet: provocation needs **plausibility**, not Greptile-grade **correctness** — a "a smart senior could plausibly suggest this" alternative, reachable from a deep flow read. But reaching even plausibility requires real analysis, so we do deep analysis internally. **This refines, not reverses, D2:** D2 governs _positioning_ — we still don't sell analysis accuracy or fight Greptile on race-detection as the product. D16 governs the _engine_ — deep analysis is an accepted internal cost that feeds capture; it's the means, never the pitch. The analysis is a **persisted, append-only artifact**: the onboarding pass builds it deeply once; future modifications append/refine it carefully rather than re-deriving wholesale (consistent with D7/D8 — expensive deep work is rare, incremental updates cheap). **Open risk:** if plausibility turns out to _also_ need accuracy-grade analysis, the wedge cracks — test early (see Open Risks). Data structure + tooling for storing/appending the analysis → deferred to technical docs. |
| **D17** | **Two-layer interception: prevent where the agent consults, catch the rest at review**                                        | The headline ("keep agents from undoing intentional design") reads as _prevention_, but MCP is **pull-based** — the agent consults Overstory only when it chooses to, and the core failure is precisely that it doesn't _know_ a topic is decision-laden, so it never asks. Default behavior is therefore catch-after, not prevent-during. Resolution = two layers on **one primitive (file → decisions)**: **(1) Guard query — best-effort prevent:** before editing, the agent asks "which decisions touch these paths?", keyed on files-in-scope (concrete, tractable), putting the _why_ in context before it picks an approach; enforced via the MCP tool description + a project rule to consult before editing (not guaranteed). **(2) Contradiction check — backstop catch:** the D11 check at PR/review catches conflicts when the agent skipped the guard query. Both layers need the same lookup — _given a code location, which decisions apply_ — which is the **reverse of the D15 pointer**. So the pointer is **bidirectional**: decision→file (navigation, D15) and file→decisions (interception, D17); build one path↔decision index, serve both ways. **Honest reframe of the promise:** "prevents where the agent consults before acting; catches the rest at review," not "the agent never errs." Index structure → technical docs. |
| **D18** | **Moat is extraction + freshness + corpus — not storage or the artifact** (repositions, does not reverse D1)                | Stress-testing surfaced two moat-shrinking forces. **(i)** The un-retrievable "why" set is **shrinking** — every new MCP connector moves PR threads / commits / tickets into the agent's reach, and models infer intent better (squeeze from both retrievability and inference). **(ii)** Overstory's output (decision + rationale + pointer) is a **commodity format identical to a CLAUDE.md / rules-file entry**, and the consumption layer is owned by the harness. So neither the stored artifact nor the serving is defensible. The honest axis is **already-externalized-anywhere vs head-only**; the only structurally uncopyable input is **head-only** rationale. This repositions Overstory as an **extraction tool, not a storage tool** — storage + MCP serving stay as necessary plumbing (does NOT reverse D1: still store decisions/rationale/pointers, never code), but the moat is (a) the capture loop's quality at extracting head-only "why" without a documentation chore (D4+D16), (b) freshness (D11), (c) accumulated repo corpus = switching cost. Two operational consequences: **(1) Seed from retrievable sources** — read commits/PRs/tickets to build analysis, draft provocations, pre-fill rationale; ask the human only for the head-only _delta_ (turns #4 from threat to asset). **(2) Artifact is commodity by design** — looks like CLAUDE.md, fine; loop quality is therefore the _existential_ metric (ties Risk 3, D16). The real competitor is **harness-native decision-memory + capture** (Anthropic/Cursor walking into this), not Greptile; defense = cross-harness neutrality + a deeper loop + corpus gravity, or accept acquired-as-feature. |
| **D19** | **Onboarding provocations are ranked + budgeted, not exhaustive**                                                             | D4 makes capture _feel_ like engagement, not a chore — but the inversion changes the _feeling_ per item, not the _count_. A real repo has hundreds of decisions; provoking on all of them at first ingestion (D8) recreates the chore as **onboarding fatigue**, and volume kills the engagement the inversion bought. So onboarding does NOT provoke on every decision. Deep analysis (D16) surfaces candidates; rank by **non-obviousness × blast radius × banned-default proximity** — skip conventional choices an agent gets right anyway; prioritize cross-cutting, high-impact decisions where a tempting _wrong_ default exists (exactly the "agent re-makes a banned choice" cases). Provoke only the **vital few**, budgeted to a tolerable session; defer the long tail to incremental passes (D8) + contradiction-triggered capture (D11). Seeding (D18) further cuts per-item effort by pre-filling rationale. Reframes onboarding from "capture everything once" to "capture the vital few, accrue the rest lazily." |
| **D20** | **Two-tier serving: always-on cross-cutting set + on-demand file-scoped guard query**                                         | D17's file→decision guard query handles relevance _once a locus is known_. But at task start the agent often has only a fuzzy goal and no files yet — and some decisions should **steer which files it even touches** (e.g. dual auth decides widget-vs-dashboard before any file is chosen) — the relevance gap D17's file-scoped lookup leaves open. Resolution = two tiers: **(1) Always-on tier** — the few highest-blast-radius, cross-cutting decisions (the top slice of D19's ranking), injected at session start regardless of task, so fuzzy-goal file selection is steered and the agent need not _know to ask_ (also partly fixes the pull-based gap, Risk 4 / D17); **(2) On-demand tier** — the D17 file→decision guard query for everything scoped to a locus, with rough-locus → decisions → corrected-locus as a feedback loop. Preserves refined context (D3): always-on is tiny, on-demand is locus-filtered — never the full set. Optional enhancement: topic-tag decisions so the always-on set can be task-targeted rather than blanket. Tier boundaries + injection mechanism → technical docs. |

## Build decisions (made entering the build phase)

These resolve items that were "still-open" and lock the technical shape. Recorded here; detailed in the technical docs (`../technical/data-model.md`, `../technical/capture-loop.md`, `../technical/serving.md`).

| #       | Decision                                                            | Why                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| ------- | ------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **D21** | **Stack: TypeScript** (confirms the assumption)                     | Dogfood repo (Kakiyo) is TS; mature MCP TypeScript SDK; one-ecosystem-first (D9). Resolves the still-open stack question.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| **D22** | **Decision scope: flow + file pointers (hybrid)**                   | Resolves the still-open decision-record-scope fork. A decision is scoped to a human-meaningful **flow** (D12 — how humans reason about architecture) and also carries file-level **pointers** (D15) to every locus it governs. Flow gives the grouping humans capture/defend against; pointers give the path→decision index the guard query + contradiction check need (D17/D20). No forced either/or — flow for humans, pointers for machines.                                                                                                                                                                                                                                          |
| **D23** | **Storage: hosted cloud service** (not in-repo files, not local DB) | A hosted backend behind the MCP server + capture UI. Strategic fit: **(a) corpus gravity** — the accumulated decision corpus is a moat pillar (D18); holding it in a service (vs exportable repo files) makes switching cost real; **(b) cross-harness neutral serving** (D18 defense) — one backend, any harness over MCP; **(c) team-ready** without re-architecture. Cost: infra + auth from day one (mild tension with D13's low-friction dogfood — mitigated by the founder using the hosted instance), and the PR contradiction check (D11) now calls the service API rather than reading committed files. _Rationale to confirm during build: corpus-gravity should be the real driver, not convenience._ |
| **D24** | **Capture runs locally (CLI), pushes results to the cloud store**   | Deep analysis + provocation run on the dev's machine against their existing checkout — no repo-access grant, no server-side clone. Only captured decisions/rationale/pointers push to the cloud (D23). Lowers the adoption trust barrier (devs resist handing a startup their private repo) and keeps clone/compute infra out of scope. Pairs with D25: execution is local, but LLM calls are metered through the backend. |
| **D25** | **LLM billing: managed** (Overstory meters + bills), via backend proxy | Overstory calls Anthropic on its own account and bills the user (not BYO-key). Cleaner UX + a real cost/revenue surface for the product. Mechanism: the local capture CLI routes LLM calls through the Overstory backend (metered proxy) — backend holds the key, counts tokens, bills. **Tension (flagged):** local execution reduces _repo-access_ exposure, but managed LLM means _code snippets still transit the backend_ for the analysis call (sent, not stored — consistent with D1). Not zero-egress; a privacy-sensitive buyer may prefer BYO-key. Acceptable for dogfood (founder's own data); revisit BYO-key as an option later. Cost: the subagent-run-cost watch-item bites from day one — you pay for Opus analysis. |
| **D26** | **Backend: Postgres + Drizzle + Hono + Better Auth**               | Postgres — relational store fits the flow→decision→pointer graph, the path↔decision join index, and multi-tenant hierarchy (D22/D17/D20/D23). Drizzle — TS-native, lightweight, typed schema. Hono — light, fast API. Better Auth — already in the founder's toolkit (Kakiyo); API keys for MCP + CLI. Host: Neon (managed PG) + Fly/Railway. |
| **D27** | **Grant Overstory repo read access (GitHub App)** — enables cloud-side capture + PR-time freshness (supersedes the local-only part of D24) | D25 (managed LLM) already sends code snippets server-side for analysis, so the privacy rationale behind D24's local-only execution is largely moot — the marginal step from "code transits for the LLM call" to "backend can read the repo" is small, and it unlocks materially more: **server-side onboarding** (connect repo, analysis runs in cloud — no _required_ local CLI), **PR-time contradiction checks** (D11) via a GitHub App, and **seeding** (D18) from full git/PR history server-side. Access is **read-only, least-privilege** (`contents:read`, `pull_requests:read`) via a GitHub App. **Credential handling:** the only long-lived secret is the GitHub App private key, held in a KMS/secret manager — never in the DB; per repo we store only the installation id and mint short-lived (~1h) installation tokens on demand. **Supersedes** the "execution is local, no repo-access grant" part of D24: capture may now run server-side; the CLI becomes an _optional_ local trigger, not the required execution site. **Resolves** the D25 egress tension by accepting egress as by-design (repo access is a superset). **New cost:** the trust barrier D24 dodged returns — repo access is a far bigger adoption ask than an API key, and we now hold repo-access credentials (security surface, see Open Risks). Trivial for dogfood (founder's own repo); the real buyer gate otherwise. |
| **D28** | **Dashboard — log every MCP call (`ServeEvent`) and visualize what the agent pulled; it doubles as the dogfood instrument** | "Visually see what the coding agent gets from MCP" is not UI candy — it's the **only honest way to measure the two unproven things** in Open Risks. MCP is pull-based (D17), so whether prevent actually fires (Risk 4 — does the agent call `guard` before editing?) and whether serve has value (Risk 3A — do stored decisions get pulled?) can't be known by argument; they must be **measured**. Mechanism: every `overstory_context/guard/check/decision/search` call writes one **`ServeEvent`** (tool, query, served decision IDs, conflict IDs, sessionId, timestamp) — the Hono serving handler does the lookup _and_ the insert in one place, so it can't be forgotten. The dashboard then serves two co-equal jobs (per build choice): **(a) Activity** — derived metrics, headlined by **consult-rate** (of runs touching decision-bearing files, what % issued a covering guard call); also coverage gaps, most-served decisions, agent-side + PR-time catches; **(b) Sessions** — per-session timeline of exactly what one agent run received, payloads reconstructed **live** from current `Decision` (IDs → render). **D1-safe:** `ServeEvent` stores decision IDs + query metadata (file paths the agent named), never code — there are no code bytes in a served payload (why+where, D15) to log. **Not a second app (D29):** Activity + Sessions are views of the same web app that hosts the capture client. Details → `../technical/dashboard.md`; entity → `../technical/data-model.md`. |
| **D30** | **Auth schema = Better Auth generated; Workspace = `organization` table; ApiKey = domain table** | Filling the schema (data-model.md) resolved how auth maps to tables. **(a)** The auth tables are **generated by the Better Auth CLI** (`@better-auth/cli generate`) into `packages/db/src/schema/auth.ts` from the `betterAuth()` config (organization plugin) — never hand-edited, regenerated when plugins/fields change. This keeps us on Better Auth's source of truth rather than a hand-rolled drift. **(b)** There is **no separate `workspace` table** — the `organization` table _is_ the workspace (D23 said "Workspace = Better Auth org"); domain tables FK to `organization.id` (workspace) and `user.id` (createdBy). **(c)** `ApiKey` is a **domain** table, not a Better Auth plugin — Better Auth 1.6 ships no apiKey plugin (the `mcp` OAuth plugin is a later option if we move off keys). Stores only `hashedKey` + `prefix`; plaintext key never persisted. Domain relations are centralized in `schema/relations.ts` (the generated auth relations stay untouched to survive regeneration). |
| **D29** | **Frontend stack: TanStack Start; Hono stays for machine clients; one web app, one shared Drizzle data layer** | The web surface (dashboard D28 + capture client) needed a frontend stack — D26 specified only the backend. Chosen: **TanStack Start** (full-stack React, type-safe routing/loaders, server functions). **Hono is not replaced** — the split is by _client kind_: TanStack Start serves the **human UI** (Activity, Sessions, Decisions, Provocations, Flows), Hono serves the **machine clients** (MCP server backend, CLI, GitHub App webhooks) that don't want a React framework. **No duplicated business logic:** both paths go through a shared `packages/db` (Drizzle) — the serving lookups + `ServeEvent` inserts are shared query functions, called by Hono (machine) and read by the web app (human). Auth: Better Auth session for the web, `ApiKey` (D26) for MCP/CLI — same backend, two credential kinds. Monorepo (pnpm/Turbo); web deploy via Nitro on Fly/Railway, same Neon Postgres. _Cost/tension noted: TanStack Start has its own server runtime, so there are two server entry points (web + Hono) — accepted, kept clean by sharing the data layer rather than the HTTP surface._ |
